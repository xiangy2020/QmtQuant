# OSkhQuant 项目编码规范

## 专项规范按需加载索引

> 以下专项规范文件按场景加载，不必每次全量读取：

| 场景 | 加载文件 |
|------|---------|
| 涉及时间/日期处理、miniQMT 数据解析、xqshare examples 脚本 | `.codebuddy/rules/data_rules.md` |
| 文件超行数限制需重构、新功能设计、分层架构调整 | `.codebuddy/rules/architecture_rules.md` |
| 开发新功能、设计新模块 | `.codebuddy/rules/workflow_rules.md` |

## 一、编码通用规范（强制）

> 每次写代码前必须主动检查以下各点，不等用户提醒。

### 1.1 网络/RPC 调用

- API 支持批量参数时，**必须批量传入**（如 `get_market_data_ex` 的 `stock_list`）
- 禁止在循环内逐条调用可批量的 API

### 1.2 文件/数据库 IO

- 能一次性读写多条的，**必须批量操作**，禁止循环单条写入

### 1.3 模块导入

- 禁止在方法体/循环体内写 `import`
- 可选依赖统一在文件顶部用 `try/except` 导入并置 `None`，方法内用 `if xxx is None` 判断

### 1.4 大文件主动检查（强制）

**每次准备向某个 `.py` 文件追加或修改代码前，必须先检查该文件的总行数：**

- 文件 **≥ 2000 行**：禁止继续追加新功能，必须先加载 `.codebuddy/rules/architecture_rules.md`，提出拆分方案，经用户确认后再动手
- 文件 **< 2000 行但新增后将突破 2000 行**：必须先停下来，提示用户该文件即将超限，给出拆分建议后再继续
- 单个 `class` **≥ 1000 行**：必须评估是否可用 Mixin 或子类拆分，给出方案后再写代码
- 单个函数/方法 **≥ 100 行**：必须拆分为多个子函数，不允许以"逻辑连贯"为由保留超长函数

> 这是写代码前的默认检查项，不是等用户指出后才做的操作。

### 1.5 平台与 xtdata 入口（强制）

- 全项目禁止直接 `from xtquant import xtdata` 或 `from xqshare import xtdata`
- 必须统一用 `from env import xtdata`
- 平台判断禁止各自调用 `platform.system()`，必须从 `env` 导入 `IS_MAC`、`IS_WINDOWS`、`IS_LINUX`、`IS_REMOTE`、`PLATFORM`

## 二、降级/兜底逻辑规范（强制）

遇到"主方案失败时是否降级到备用方案"的决策点，**禁止自行决定降级策略**，必须先停下来询问用户确认，例如：

- 交易日历获取失败时是否反推
- 数据源不可用时是否切换备用源
- 缓存失效时是否用旧数据兜底

---

## 三、业务逻辑不确定时规范（强制）

遇到业务字段特殊值含义、API 返回格式、业务规则等不确定的问题时，**必须先问用户，不能自己猜测或假设**。

---

## 四、新功能开发流程规范（强制）

### 4.1 后端优先开发流程

**新功能开发必须遵循以下流程：**

```
需求分析
   ↓
设计后端服务接口（Layer 2）
   ↓
实现纯 Python 后端逻辑
   ↓
编写单元测试验证后端功能
   ↓
集成测试
```

### 4.2 禁止跳过后端直接开发 CLI/接口

- 禁止在没有后端服务接口的情况下直接写 CLI 业务逻辑
- 禁止在 CLI 层直接调用底层 API（必须通过服务层）

### 4.3 代码审查要点

审查新代码时重点关注：
- 是否遵循分层架构原则
- CLI 层是否只负责参数收集和结果展示
- 后端服务是否独立、可测试
- 回调接口是否规范统一
