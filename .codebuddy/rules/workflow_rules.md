# 开发工作流规范

> 适用场景：开发新功能、设计新模块时加载本文件。

---

## 一、新功能开发流程规范（强制）

### 1.1 后端优先开发流程

**新功能开发必须遵循以下流程：**

```
需求分析
   ↓
设计后端服务接口（Layer 2）
   ↓
实现纯 Python 后端逻辑
   ↓
编写单元测试验证后端功能
   ↓
集成测试
```

### 1.2 禁止跳过后端直接开发 CLI/接口

- 禁止在没有后端服务接口的情况下直接写 CLI 业务逻辑
- 禁止在 CLI 层直接调用底层 API（必须通过服务层）

### 1.3 代码审查要点

审查新代码时重点关注：
- 是否遵循分层架构原则（详见 `.codebuddy/rules/architecture_rules.md`）
- CLI 层是否只负责参数收集和结果展示
- 后端服务是否独立、可测试
- 回调接口是否规范统一

## 二、功能开发后文档同步规范（强制）

### 2.1 触发条件

以下任意一种情况发生后，**必须**同步更新对应文档，不允许以"后续再补"为由跳过：

| 触发条件 | 必须更新的文档 |
|---------|--------------|
| 新增后端服务方法（`DataService`、`BacktestService` 等） | 对应服务类的接口说明 |
| 新增或修改 CLI 功能模块 | 相关模块的使用指南文档 |
| 新增或修改数据字段、API 参数 | 相关模块的字段说明注释或文档 |
| 新增配置项（`.env`、`config.json` 等） | 配置项说明文档或 README |
| 修改分层架构（新增 Layer、调整模块职责） | `.codebuddy/rules/architecture_rules.md` |
| 新增数据处理规则（时间戳格式、集合比较等） | `.codebuddy/rules/data_rules.md` |

### 2.2 文档同步流程

功能开发完成后，**在提交代码前**必须执行以下检查：

```
功能代码完成
   ↓
逐条对照 2.1 触发条件，判断哪些文档需要更新
   ↓
更新对应文档（不允许跳过）
   ↓
提交代码（文档与代码同一次提交）
```

### 2.3 AI 辅助开发时的强制行为

**当 AI（本助手）完成功能代码后，必须主动执行以下动作，不等用户提醒：**

1. **自动检查触发条件**：对照 2.1 表格，逐条判断本次改动是否命中
2. **主动提示需更新的文档**：在回复末尾列出"本次需同步更新的文档"清单
3. **询问是否立即更新**：如用户确认，立即生成文档更新内容
4. **禁止沉默跳过**：不允许完成代码后不提文档，即使用户没有主动要求

**提示格式模板（每次功能完成后必须输出）：**

```
📄 文档同步检查：
- [ ] 相关服务类的接口说明文档
- [ ] .codebuddy/rules/xxx_rules.md → [受影响规则]
- [ ] 其他：[具体文档路径]

是否现在更新以上文档？
```

### 2.4 文档内容规范

更新文档时必须包含以下内容：

- **功能描述**：一句话说明该功能做什么
- **接口/参数说明**：方法签名、参数类型、返回值
- **使用示例**：至少一个调用示例（代码片段）
- **注意事项**：已知限制、特殊处理逻辑、与其他模块的依赖关系

### 2.5 禁止事项

- **禁止**：功能上线后文档为空或仍是旧版本描述
- **禁止**：以"代码即文档"为由不写任何说明
- **禁止**：文档只写"TODO"或"待补充"后提交
- **禁止**：AI 完成代码后不主动提示文档同步需求
